# Перенос приложения на Vite

Данная версия проекта была перенесена на сборку React + Vite для удобства использования и дальнейшей работы над проектом если это будет необходимо.  


# О проекте
Данный учебный проект - реальный кейс от компании «СКАН».

Компания «СКАН» разработала новый API для поиска публикаций о компании (юридическом лице) в средствах массовой информации по ИНН. Целевая аудитория — бизнес и ИП, которым нужно проверить добросовестность подрядчиков на основе публичных данных.


Фронтенд-часть сервиса получает данные с бэкенда, то есть постоянно работает с API. Именно поэтому взаимодействие с API — ключевая часть кейса.

Серверная часть приложения уже готова, основная задача этого проекта — разработать клиентскую часть приложения.

# Описание проекта

### Общие требования/рекомендации к проекту:
  - Проект должен был выполнен на React
  - Интерфейс должен быть поделён на компоненты. Деление должно быть логичным и оправданным
  - Проект будет работать со множеством данных. Рекомендуется использовать более продвинутый инструмент хранения и изменения данных, чем обычный state (В данном проекте используется Redux)
  - При написании кода стараться следовать принципам KISS,DRY
  - Нет ограничений на использование любых инструментов и дополнительных библиотек, но нужно чтобы их применение было оправдано и не усложняло код без необходимости.
  - Вёрстка должна соответствовать макету. Добиваться Pixel Perfect соответствия не обязательно, но основные моменты должны быть соблюдены
  - Используйте любой вариант подключения стилей на ваше усмотрение
  - Приложение должно корректно отображаться и работать на мобильных устройствах (только для мобильных устройств с конкретным разрешением в 375px как в макете)

#### Клиентская часть сервиса
Клиентская часть приложения состоит из:
  - главной страницы, которая содержит описание сервиса и доступна всем пользователям без авторизации. 
  - страницы с формой авторизации
  - страницы с формой для ввода параметров запроса, которая доступна только авторизованным пользователям
  - страницы с выводом результатов запроса, содержащую карусель для горизонтального переключения общей сводки по количеству публикаций и рискам, а также список публикаций, подгружающиеся через "lazy loading"

<br>

### Краткий сценарий основной работы приложения:

  1. Открываем приложение (без авторизации) и попадаем на главную страницу (это единственная страница, доступная неавторизованному пользователю) 
  2. Используя форму авторизации, заходим в аккаунт 
  3. После успешной авторизации в фоновом режиме запрашиваются данные об аккаунте пользователя и выводятся в шапку страницы
  4. Переходим на страницу поиска и задаём параметры. Если параметры поиска введены корректно, получаем сводную информацию по количеству публикаций и рискам и выводим ее в виде таблицы на странице результатов
  5. Параллельно, используя те же параметры, выполняем поиск непосредственно самих публикаций
  6. После успешного завершения запроса получаем список ID публикаций и запрашиваем содержимое первых 10 публикаций
  7. Если найденных публикаций больше 10, нажимаем на кнопку «Показать больше» и подгружаем следующие 10 публикаций

Список аккаунтов для авторизации - [https://docs.google.com/spreadsheets/d/16sXmrKf--J-5F7AMwMNCpRVZovQmTsvF_tcDoFm1en8/edit?gid=1906748092#gid=1906748092]

### Что реализовано в проекте и какие инструменты использовались
<br>

1. Сверстан интерфейс приложения для мобильной и десктоп версий
2. Реализован роутинг и разделение прав (все страницы, кроме главной, доступны только зарегистрированному пользователю)
3. Добавлена JWT авторизация пользователя ("Access"-токены со сроком годности без refresh-токенов)
4. Внедрен Redux(Redux Toolkit) для хранения состояний проекта
5. Подключена бэкенд-часть к фронтенду, настроена отправка запросов для авторизации, параметров поиска и вывода необходимых данных пользователю на страницу.
6. Добавлены и отлажены детали для гладкого UX - лоадеры, "lazy loading", валидация форм, вывод ошибок и оповещений и т.п.

При проверке приложения я постарался по максимум отловить все ошибки и при открытии и использования проекта сбоев не наблюдалось. См. пункт об ошибках ниже для доп. информации

